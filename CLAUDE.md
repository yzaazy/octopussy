# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A static Astro site that tracks which LEGO parts of the "Octopussy" model the user owns. The three BrickLink wanted-list XML files in the repo root (`Octopussy.xml`, `backwall octopussy.xml`, `Octopussy power.xml`) are the source of truth — they are parsed at build time, never modified by the app, and can be replaced with fresh BrickLink exports at any time.

## Commands

```sh
npm run dev           # dev server at http://localhost:4321
npm run build         # static build to dist/ (prebuild downloads missing part images)
npm run preview       # serve the built site
npm run fetch-images  # download part images to public/parts/ (idempotent, skips existing)
npm run fetch-names   # regenerate src/data/part-names.json from Rebrickable's parts DB
```

After changing a wanted-list XML: run `fetch-names` (the JSON is committed; normal builds don't fetch it) — `fetch-images` runs automatically via prebuild.

There are no tests or linters configured.

## Architecture

- `src/lib/parseWantedList.ts` — parses a BrickLink wanted-list XML at build time into `{ key, partId, colorId, qtyNeeded }`, merging duplicate part+color entries. Item key format: `${partId}-${colorId}`; the client prefixes it with the section slug (`octopussy:3003-3`) because the same part can be needed by more than one list.
- `src/lib/colors.ts` — static BrickLink color ID → name/hex map covering only the colors used in the XMLs. If a new XML introduces an unknown color ID, add it here (values from bricklink.com/catalogColors.asp).
- `src/lib/partNames.ts` + `src/data/part-names.json` — part id → English name, generated by `scripts/fetch-names.mjs` from Rebrickable's free parts CSV (with BL→RB alias fixups and a manual list for BrickLink-only ids). Search matches names as well as part numbers.
- `src/pages/index.astro` — thin page: parses the XMLs, composes the components below, loads `scripts/main`. No UI framework anywhere — vanilla TS over DOM data-attributes.
- `src/components/` — `SiteHeader.astro` (masthead, sync indicator, stud bar, filter controls), `PartSection.astro` (collapsible `<details>` section), `PartCard.astro` (one part; carries the `data-key/needed/part/color` attributes the scripts rely on), `SiteFooter.astro` (download/backup buttons, build stamp).
- `src/scripts/` — client modules, `main.ts` is the only entry: `state.ts` (progress state, server GET/PUT + localStorage mirror, sync indicator), `totals.ts` (card/counter/bar rendering, section-complete detection), `filters.ts` (search/kleur/nog-nodig + sorting), `collapse.ts` (remembered section collapse), `lights.ts` (kermis light program switcher + inloop sequencer), `confetti.ts`, `backup.ts` (missing-parts BrickLink XML, JSON backup import/export).
- `src/styles/global.css` — ALL styling, deliberately unscoped (imported once in index.astro): the scripts toggle classes and create elements dynamically (confetti), which Astro-scoped styles would not match.
- `scripts/fetch-images.mjs` (run by `prebuild`) downloads part images from BrickLink's CDN (`https://img.bricklink.com/ItemImage/PN/{colorId}/{partId}.png`) into `public/parts/{colorId}-{partId}.png`, which is gitignored. Cards use the local image with a two-step `onerror` fallback: CDN color image, then colorless `PL/{partId}.png`. Node 24+ is assumed (the script imports the `.ts` parser directly via type stripping).

### Persistence model

Owned quantities live in a single flat object `{ [sectionKey]: ownedQty }`:
- Loaded from `GET progress.json` (same origin); falls back to `localStorage` if that fails (which is the normal case in `npm run dev`).
- Saved via debounced `PUT progress.json` — in production nginx's dav module writes the file (see `deploy/nginx.conf`); `localStorage` is always mirrored as backup. The sync indicator in the header reflects which mode is active.
- `progress.json` is served from and written to `/data` (a mounted volume), NOT the site root, so image updates never clobber progress. It's created by the first PUT.

## Deployment

Push to `main` → GitHub Actions (`.github/workflows/docker.yml`) builds the multi-stage `Dockerfile` (node build → nginx:alpine with `dist/` and `deploy/nginx.conf` baked in) and publishes `ghcr.io/yzaazy/octopussy:latest`. TrueNAS runs it as a Custom App with one host-path mount: a dataset (writable by uid 101) at `/data` for `progress.json`. Auth is handled by Pangolin in front — nothing in this app. Full steps in README.md.
